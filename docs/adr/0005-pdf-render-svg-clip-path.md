# PDF render: diagonal P10/P11 pair via CSS `clip-path: polygon(...)`, not SVG `<clipPath>` or baked-alpha PNGs

The legacy WeasyPrint pipeline renders the page-3 P10/P11 diagonal split by baking trapezoid alpha masks into two intermediate PNGs (`_diag_left.png`, `_diag_right.png`) at render time via PIL, then overlaying them in the HTML with the cream background showing through the diagonal gap. The iOS PDF renderer does the split entirely in markup: two `<img>` tags share one figure, each clipped to a trapezoid via CSS `clip-path: polygon(...)`. The ~3pp horizontal offset between the left polygon's right edge and the right polygon's left edge lets the cream container background show through as the cream gap between trapezoids, matching the 0.16in gutter elsewhere on the page. No image processing on device, no intermediate PNGs written to disk.

The first iOS implementation tried inline SVG with `<clipPath>` polygons (one `<svg viewBox="0 0 100 100" preserveAspectRatio="none">` containing two clipped `<image>`s and a `<line>` for the seam). It device-verified on 2026-05-28 with two defects (issue #24): (1) `preserveAspectRatio="none"` stretches the viewBox non-uniformly to fit the ~2.17:1 cell, which distorts the clipped images horizontally; (2) both polygons share the diagonal `(70,0)→(30,100)` literally, leaving zero cream gap at the seam. CSS `clip-path: polygon(...)` percentages adapt to the container's aspect ratio without viewBox math, and giving the two polygons a small horizontal offset reintroduces the cream gap without an explicit stroke. WebKit on iOS supports `clip-path: polygon(...)` natively in `WKWebView.createPDF(configuration:)` output.

The legacy PIL-baked-PNG approach existed because WeasyPrint 68's SVG `<clipPath>` support was incomplete. WebKit doesn't share that limitation, and CSS `clip-path` removes the need for an SVG wrapper entirely. If the CSS approach itself fails some future device-verify (geometric drift in WebKit's PDF output, paint-order surprises with the figcaptions, etc.), the fallback is the legacy PIL approach ported to CoreGraphics — write `_diag_left.png` and `_diag_right.png` into the player directory alongside the source panels before rendering. The CSS version is small enough (one short stylesheet block plus two `<img>` tags) to throw away if it fails. The CoreGraphics fallback is a meaningfully larger slice — order of magnitude more code, more failure surface — which is why it isn't the default.
