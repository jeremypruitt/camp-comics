# PDF render: diagonal P10/P11 pair via inline SVG `<clipPath>`, not baked-alpha PNGs

The legacy WeasyPrint pipeline renders the page-3 P10/P11 diagonal split by baking trapezoid alpha masks into two intermediate PNGs (`_diag_left.png`, `_diag_right.png`) at render time via PIL, then overlaying them in the HTML with the cream background showing through the diagonal gap. The iOS PDF renderer does the same split entirely in markup: each panel's source PNG goes into an `<image>` inside an inline SVG `<clipPath>` polygon, and a sibling `<line>` paints the 1pt diagonal stroke. No image processing on device, no intermediate PNGs written to disk.

The legacy approach existed because WeasyPrint 68's SVG `<clipPath>` support was incomplete, forcing PIL to do the masking instead — the layout comment in `_legacy/layout/comic.css` calls this out explicitly. WebKit on iOS doesn't share that limitation: `<clipPath>` renders correctly in `WKWebView.createPDF(configuration:)` output. Pushing the mask into CoreGraphics on iOS would add an image-processing layer to the renderer — one more failure mode, one more thing to test, intermediate files to clean up — for fidelity that the spike's eyeball test against an existing `.done` player will tell us we don't need.

If the spike's PDF shows divergence at the diagonal seam (misalignment between the two trapezoids, stroke artifacts, or a border mismatch with the surrounding rectangular panels on page 3), the fallback is the legacy PIL approach ported to CoreGraphics — write `_diag_left.png` and `_diag_right.png` into the player directory alongside the source panels before rendering. The SVG version is small enough (one `<svg>` block in the page-3 markup, ~30 lines) to throw away if it fails the eyeball test. The CoreGraphics fallback is a meaningfully larger slice — order of magnitude more code, more failure surface — which is why it isn't the default.
