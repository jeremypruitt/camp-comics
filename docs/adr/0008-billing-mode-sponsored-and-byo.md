# `BillingMode` is BYO-primary with a 2-comic Sponsored trial, gated by per-comic generation budget and global GCP cap

The "who pays Google" axis is named **`BillingMode`** with two cases: `.sponsored` (the `comic-harness` Firebase project pays Jeremy's card) and `.byo` (the user pastes their own Gemini Developer API key; calls bill against their Google account, uncapped). The axis is stored per-app in `UserDefaults`; the BYO key lives in Keychain. It is not recorded per-player — flipping mid-comic is supported because both modes call `gemini-2.5-flash-image` with the same `STYLE_SUFFIX` ([[adr-0004]]), so STYLE_SUFFIX continuity is unaffected by the switch.

The first-launch posture is **BYO-primary**: the consent modal frames the app as "Try 2 free, then bring your own key." This is the lowest-rejection-risk posture under App Store review — the reviewer sees the BYO model immediately, BYO is positioned as the product, and Sponsored reads as the demo. The riskier alternative (Sponsored-default with BYO surfaced only at cap-hit) reads to a reviewer like the developer's paid service ran out and is redirecting payment externally — the IAP-circumvention shape Apple rejects. The pure-Sponsored alternative (no BYO at all, cap-hit means "come back next month") was rejected because it loses the uncapped-enthusiast and the cohort-mode use cases.

Three friction controls work together; the third is the only one that bounds Jeremy's liability:

1. **Per-comic generation budget** — `2 × slots` = 32 Vertex calls per Sponsored comic. Every `gemini-2.5-flash-image` call decrements it: initial generation, Re-roll, Re-prompt, and Re-roll-after-accept all count uniformly. The QA-gate generation does NOT count (it gates whether a comic starts at all). Counter visible mid-flight as a low-key chip ("23 re-rolls left"). On exhaustion, a modal offers "accept current candidates and finalize" or "paste a Gemini key to continue in BYO." Caps per-comic spend at ~$1.25.
2. **Per-install Sponsored trial** — 2 finalized comics per install, where "finalized" means the comic PDF has been generated. Incomplete comics don't decrement. Keyed on the Firebase Auth anonymous UID + server-side Firestore counter, so the counter survives reinstall.
3. **Global GCP budget cap** — hard budget limit on the `comic-harness` project. LOAD-BEARING. Per-install and per-comit caps are friction controls layered above; only the GCP cap actually stops billing in extremis. Must be configured before any Sponsored code ships.

Worst-case Sponsored spend per install = 2 × $1.25 = ~$2.50. At a $50/month global cap, that's ~20 maxed-out installs/month worst-case; median use (median panel ≈ 1.5 candidates → ~$0.60/comic) puts the realistic ceiling closer to ~40 installs/month. Cohort runs (~60 players/week) operate in `.byo` with Jeremy's own GCP key and don't touch the Sponsored counter.

**Content-policy disclosure** is a first-launch blocking consent modal naming Google as the generation provider, with a permanent "About AI generation" entry in Settings and a long-press "Report image" action sheet on every accepted panel. Settings-only disclosure was rejected — 2025–2026 App Store review increasingly demands a visible consent flow for AI image generation apps, especially those generating from a child's likeness. Per-generation banners were rejected as operationally noisy for cohort runs (Jeremy generating 60 players would see the banner 1,000+ times).

Implementation order is non-negotiable: (1) set the GCP global cap manually in the console — no code, no agent; (2) register a Gemini Developer API key against `comic-harness` and re-spike `.googleAI()` returns images; (3) wire Firebase Auth anonymous + Firestore counter; (4) build first-launch consent + Settings entry; (5) build the cap-hit modal + BYO paste UI + Keychain storage; (6) wire the per-comic generation budget chip + exhaustion modal. Steps 1 and 2 are the only ones that can't be undone by reverting code, so they ship first.
