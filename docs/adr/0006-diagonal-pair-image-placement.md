# Diagonal P10/P11 pair: image placement via prompt-driven composition + Apple Vision saliency safety net

The diagonal pair from ADR-0005 (CSS clip-path trapezoids over two 16:9 panel sources) frames the subject correctly only when the generated subject actually sits in the visible region of its trapezoid. Under `object-fit: cover` + `object-position: center top`, a 16:9 source rendered into the 6.025×2.67in cell scales to 100% wide × 127% tall — there is zero horizontal overflow, so `object-position-X` cannot recompose the subject after the fact. The subject lands where the model put it, which on field test (slice 30a device verify, 2026-05-29) is approximately centered regardless of prompt pressure, because the reference photos already anchor the model on a centered subject. Two layered interventions: prompt-driven composition first (slice 30a / `PromptBuilder.panelLayoutHints`), Apple Vision attention-based saliency + CSS `transform: scale + translate` second (slice 30b / `SaliencyAnalyzer` + `HTMLAssembler.PanelPosition`). The prompt hint is cheap and durable across model upgrades; the saliency transform is deterministic and recovers placement even when the model ignores composition instructions entirely. They compose cleanly — the saliency layer reads whatever the model produced and corrects from there.

The geometry: cell aspect 6.025/2.67 = 2.258, source aspect 16/9 = 1.778, source-display-over-cell-height = 2.258/1.778 = 1.270 (so source displays 127% of cell height, upper 78.7% visible under top-aligned cover). Polygon centroids in cell coordinates, computed via the shoelace formula on the clip-path vertices: `diag-left` (vertices at (0,0), (68.25%,0), (28.25%,100%), (0,100%)) ≈ (25.5%, 43.1%); `diag-right` (vertices at (71.75%,0), (100%,0), (100%,100%), (31.75%,100%)) ≈ (74.5%, 56.9%). The right trapezoid's centroid sits lower than the left's because it is wider at the bottom — not a vertical mirror. `PanelPosition(xPct, yPct)` is the salient subject centroid in 0–100 source coordinates with top-left origin. `HTMLAssembler.diagTransformAttr` converts the source centroid into a CSS `transform: translate(tx%, ty%) scale(1.4)` such that, after the scale-around-center and translate, the source centroid lands on the polygon centroid in cell coords. The fixed scale of 1.4 gives ±20% headroom on both axes — enough to recompose any reasonable centroid without exposing the cream container background through the trapezoid edge.

The map is per-panel and the `assemble(panelPositions:)` default is empty, which falls through to the slice-30a CSS baseline byte-identically (regression-guarded). `PDFRenderer.diagonalPairSaliency` runs Vision on staged `panel_10.png` + `panel_11.png` in parallel before HTML assembly; failures, missing files, or empty salient regions silently omit that panel from the map. We deliberately do not cache the saliency result to disk — recompute is on the order of 100ms per panel against the multi-second WKWebView PDF render, invisible at the user level — and the saliency call is scoped strictly to P10/P11 today. If a future layout introduces another clipped panel that needs runtime placement, the API surface is already in place: add an entry to the map and a target centroid to `diagTransformAttr`. The opposite extreme — saliency on every panel — is out of scope; the other 10 panels render full-bleed into rectangles where `object-position` already handles the slack.

If both layers fail in the field (Vision returns garbage centroids on stylized D&D illustrations, or the scale-and-translate produces visually awkward overzoom), the fallback is `panelPositions: [:]` at the call site — a one-line revert to the slice-30a baseline that ships a centered crop, which is the same failure mode the diagonal pair had before this ADR.
